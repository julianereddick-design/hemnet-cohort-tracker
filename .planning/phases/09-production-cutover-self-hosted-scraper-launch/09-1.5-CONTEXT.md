<decisions>
## Implementation Decisions

### Root-cause hypothesis (D-13)

- **D-13:** Plan 09-1.5 **locks the hypothesis** that VERF-09-1 carry-forward issues #1 (Oxylabs detail-page rejection) and #2 (county-loop bail after Stockholm) share **one** Oxylabs-side root cause. Evidence from `verf09-1-logs/wet-run.log`:
  - 16 oxylabs-fallback-success entries (Stockholm `/sok/till-salu` pages 1–16, lines 9–99), all via Oxylabs (`directSuccessCount=0`).
  - Stockholm page 17 (line 105) and VG/Skåne/Uppsala page 1 (lines 114, 122, 130) all return `OXYLABS_API_NON_200` — the Oxylabs API itself, not the target Booli page (per `lib/scrape-http.js:140-145`).
  - Every `/bostad/*` detail call (319 of them) returns the same `OXYLABS_API_NON_200`.
  - The "county-loop bail" is therefore **not** a state leak in `walkCountySearch` — it's the same Oxylabs rejection that kills the detail fetches, just kicking in after ~16 successful search calls.

  Planner writes a single fix path. If the wet-run shows the hypothesis is wrong (one symptom resolves but the other persists), **re-plan** rather than back-filling tasks.

### Oxylabs subscription scoping (D-14)

- **D-14:** The previous Oxylabs access was a **trial account that has effectively expired** (user-confirmed). The 16-successes-then-100%-failure signature in the wet-run is the textbook trial-credit-exhausted pattern. **Plan 09-1.5 will subscribe to a paid Oxylabs plan** — no point hunting for "in-code only" fixes.

- **D-14a:** Plan choice approach: **planner produces a shortlist; user picks**. The plan must include a research/scoping task that:
  1. **Estimates per-cycle Oxylabs volume** across all 4 cron jobs (Job A hemnet-targeted-refresh, Job B hemnet-weekly-seeding, Job C booli-targeted-discovery, Job D booli-targeted-refresh). Use the wet-run as anchor (339 calls for one Job C cycle); compute per-month projection at the every-2-days cadence locked in `09-CONTEXT.md` D-06.
  2. **Researches Oxylabs paid plans** — Web Scraper API tiers (universal vs. web_scraper-with-rendering), Residential Proxies, SERP/E-commerce. Surface monthly cost, request quota, features (rendering, premium pool, JS-heavy site support).
  3. **Produces a 2–3 option shortlist** with monthly cost + tradeoffs. User reviews mid-execute, picks one, planner proceeds with fix + wet-run against the chosen plan's credentials.

- **D-14b:** **Wet-run gates on user having an active paid subscription with new credentials in `.env`**. The planner should design VERF-09-1.5 such that the fix-validation step is the FIRST live use of the new credentials — if the paid plan still doesn't fix it, the plan re-opens before `09-02` starts.

- **D-14c:** Open to: switching `source` (universal → web_scraper), enabling `render: 'html'`, premium pool, or other Oxylabs Web Scraper API features that the paid plan unlocks. Stay on Oxylabs (Web Scraper API or another Oxylabs product) — vendor swap is **out of scope** for 09-1.5. If Oxylabs paid plan still doesn't fix it after the shortlist iteration, re-open the plan and reconsider vendor.

## Existing Code Insights

### Reusable Assets

- **`lib/scrape-http.js#fetchViaOxylabs`** is the single point of Oxylabs invocation. Any per-endpoint config change (search vs. detail) lives here. The current body shape (`source: 'universal'`, `geo_location: 'Sweden'`, `user_agent_type: 'desktop'`) is the smallest viable Oxylabs call — paid plans likely add `render`, `premium`, or per-source flags that this object can accept verbatim.
- **`lib/scrape-http.js#getOxylabsStats` + `resetOxylabsStats`** — module-level counters (`oxylabsCallCount`, `oxylabsFailureCount`, `directSuccessCount`, `oxylabsFallbackRate`). Already wired into Job C's summary; the 09-1.5 wet-run uses these unchanged.
- **Plan 09-01 hardened worker-pool + `Final:` emitter** (commits `721ea52` + `a25c1f0`) — Job C will preserve `workerErrors:0`, `budgetExceeded:false`, `EXIT=0` semantics through 09-1.5 regardless of what changes in the Oxylabs config.

### Established Patterns

- **Diagnostic doc + investigation entry points**: Plan 09-01's Task 1 wrote `scripts/diagnose-verf-b2.md` as a root-cause analysis BEFORE the fix. Optional but encouraged for 09-1.5.
- **Probe-script gating**: Plan 09-01's VERF-09-1 ran the full ~15-min wet-run as the verification gate. With paid Oxylabs credits, a smaller probe (10–20 URLs spanning search + detail) is the cheaper feedback loop. Match `scripts/probe-fetch-fallback.js` shape for the probe.
- **SCRAPE_FORCE_OXYLABS=1 env flag** (`lib/scrape-http.js:40-42`) — exists exactly for this kind of debugging. Probe scripts can force the Oxylabs path bypass even when direct curl works.

### Integration Points

- **`lib/scrape-http.js` is the entry point for all four Booli/Hemnet jobs** (Jobs A, B, C, D). Any change here ripples to every cron job. The 09-1.5 fix MUST preserve the existing public surface (`getWithRetry`, `extractNextData`, `getOxylabsStats`, `resetOxylabsStats`) — Plans 09-02 / 09-03 / 09-04 do not need to know about the change.
- **`.env` credential rotation**: switching from trial to paid Oxylabs likely changes username/password. Deploy-side coordination is implicit — `deploy-instructions.md` already documents the existing creds; the cutover step needs to swap them.
- **Wet-run cost**: Plan 09-01 wet-run consumed 339 Oxylabs calls (mostly failures, so likely "free" on trial). On the paid plan, each wet-run costs real money. The planner must scope wet-run frequency accordingly — probe-first, full-wet-run-once-confident.

## Specific Ideas

- **Log evidence priority**: `verf09-1-logs/wet-run.log` is the single source of truth for the failure signature. The planner's first move should be to re-read it (especially lines 9–135 for the failure-mode transition and lines 105, 114, 122, 130 for the per-attempt timings) rather than re-running the wet-run.

- **The 4-second-per-county pattern** (lines 108, 117, 126, 135 timestamps: `22:10:44.654`, `22:10:48.891`, `22:10:53.161`, `22:10:57.293`) is the direct-curl 3-attempt retry envelope (1s + 2s + 4s with jitter ≈ 4s wall-clock). Once the planner sees this, it's strong support for the "Oxylabs-side rejection, NOT walkCounty state leak" reading in D-13.

- **`OXYLABS_API_NON_200` is the API-layer error**, not the target-page error. The `body` slice in the error message at `lib/scrape-http.js:142-143` should be inspected — the actual Oxylabs API response body (first 300 chars) will tell us if it's a 402 Payment Required, 429 Quota Exceeded, 403 Forbidden, or something else. The planner should grep the wet-run log for the `body=` substring or instrument the probe script to log it explicitly.
