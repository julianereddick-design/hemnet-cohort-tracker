## Thread 1 — How the discussion started (Plan 09-02 Task 5 wet-run scope)

**Trigger:** Operator paused at Plan 09-02's Task 5 wet-run gate. Asked whether the existing Booli cohort finder works and whether the Hemnet matching process can be trusted.

**Operator's framing:** "I'm worried about the Hemnet matching process - can you walk me through how this currently plans to work."

**Claude's initial analysis:** Walked cohort-create's matching SQL — `postcode = $1 AND LOWER(TRIM(street_address)) = LOWER(TRIM($2)) AND listed BETWEEN $3 - 7d AND $3 + 7d AND county = ANY([...])`. Listed four failure-mode hypotheses:
1. Postcode format mismatch (Booli's own comment in `hemnet-targeted-match.js:144-148` notes Booli stores '176 71' vs Hemnet '17671'; cohort-create doesn't normalize)
2. Street_address vs Booli's title differing on apt suffixes, building letters
3. ±7 day listed window vs differing date semantics
4. Job B coverage gaps (Hemnet listing exists but Job B's 15-page search didn't find it)

**Operator pushback:** "Why do you think its not in Job B - this is where the issue will be. We need to FIND the right listings on Hemnet, we don't have them all."

**Key insight from operator:** cohort-create is downstream — it can only match what's already in `hemnet_listingv2`. Job B is the seeding gate. Even with a perfect cohort-create SQL, a missing Hemnet row can never be matched. The operator's intuition was that Job B is the primary loss point.

**Decision implication:** Job B is the focal point of the matching fix. cohort-create SQL fixes are secondary and deferred.

## Thread 7 — Hemnet's URL vocabulary verification

**Operator instruction:** "you test them"

**Live probe:** Fetched `hemnet.se/bostader?item_types[]=<token>` for each candidate token, parsed the `housingFormGroups` value from Hemnet's GraphQL search args in the Apollo state. Tokens producing a non-empty `housingFormGroups` are VALID; tokens producing empty `housingFormGroups` are silently IGNORED by Hemnet (returning all listings).

**Verified valid tokens (6):** `bostadsratt → APARTMENTS`, `villa → HOUSES`, `radhus → ROW_HOUSES`, `fritidshus → VACATION_HOMES`, `gard → HOMESTEADS`, `tomt → PLOTS`.

**Verified IGNORED tokens:** `kedjehus`, `parhus`, `lagenhet`, `andelsbostad`, `agarlagenhet`, `tomt-mark` (Claude initial guess; correct is `tomt`), `gard-skog` (Claude initial guess; correct is `gard`).

**Mapping decisions:**
- Lägenhet → `bostadsratt` (1:1 semantic)
- Villa → `villa` (1:1)
- Radhus → `radhus` (1:1)
- Fritidshus → `fritidshus` (1:1)
- Tomt/Mark → `tomt` (Booli adds "Mark", Hemnet uses bare "tomt")
- Gård → `gard` (1:1)
- **Kedjehus → `villa`** (Hemnet bundles into HOUSES — no separate filter)
- **Parhus → `villa`** (Hemnet bundles into HOUSES — no separate filter)

**Decision implication: D-25 locks the 8-entry mapping table.** 100% of 140 sampled Booli types have a defined mapping target. Implementation in `lib/booli-to-hemnet-mapping.js`.

## Thread 9 — COALESCE-preserve write semantics

**Trigger:** Designing Job C's ON CONFLICT UPDATE clause.

**Question:** When a refresh sees a null value for `price` (Apollo state was briefly malformed), should it overwrite the existing good value with null?

**Decision:** No. Use `COALESCE(EXCLUDED.x, booli_listing.x)` — only overwrite when the new value is non-null. Same pattern in Job D's UPDATE.

**Rationale:** A null parse can be a transient parser failure. Persisting it would silently degrade the data. The cost of preserving stale values is bounded (Job D refreshes every 2 days, so values get re-evaluated frequently); the cost of clobbering valid data is unbounded (Metabase reports could lose key fields on legacy rows).

**Decision implication: D-24 COALESCE-preserve.**
