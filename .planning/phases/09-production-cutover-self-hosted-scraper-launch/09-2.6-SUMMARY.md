## Why the cohort match rate (47.7%) is lower than Job B's match rate (60.8%)

Job B's `matchedFromSearch` counts any Booli row where Hemnet's narrowed search returned a card with matching street + ±7d listed date. The cohort-create.js join is stricter — exact postcode match AND `LOWER(TRIM(street_address))` equality AND ±7d. The ~13 percentage-point gap reflects the postcode-mismatch discount documented in STATE.md 09-2.5 #7 (same property on Booli vs Hemnet sometimes carries different postcodes due to upstream data drift; treated as conservative-by-design until Phase 10 decides whether to loosen).

## The two new Phase 10 carry-forwards opened by this plan

**09-2.6 #1 — cron-wrapper.js missing signal handlers (SIGHUP / SIGTERM / SIGINT).** Discovered live during the W20 recovery: operator kicked off `hemnet-targeted-match.js` via DO web console without nohup; SIGHUP on disconnect killed node; cron_job_log row 418 stayed on `status='running'` forever because cron-wrapper.js:79-80 only registers `uncaughtException` + `unhandledRejection`. Required a manual SQL UPDATE to unstick (`scripts/unstick-cron-row-418.js`). Same gap will fire on ANY operator-kill of a running cron — and on the original 09-2.6 plan's Task 5 step 2 (operator-kill-then-restart for the W20 recovery, which we didn't end up needing because the original job was already dead). Phase 10 fix: ~5 lines adding `process.on('SIGHUP', handleFatal)`, `process.on('SIGTERM', handleFatal)`, `process.on('SIGINT', handleFatal)` alongside the existing two, with `handleFatal` resolving status to `'killed'` for signals vs `'failure'` for genuine exceptions.

**09-2.6 #2 — hemnet_listingv2 functional index for delta filter fallback path.** Plan 09-2.6 Task 2 measured the full delta filter (`bl.crawled > X OR NOT EXISTS hemnet_listingv2`) at 104s EXPLAIN — too slow for the 120s statement_timeout. Fell back to crawled-only (93ms). Root cause: the NOT EXISTS subquery did a seqscan of hemnet_listingv2 (~129k rows) 2,591 times because there's no functional index on `LOWER(TRIM(street_address))`. Phase 10 fix: `CREATE INDEX CONCURRENTLY hemnet_listingv2_norm_street_idx ON hemnet_listingv2 (LOWER(TRIM(street_address)))`. Unlocks the fuller delta filter, which would recover the ~30% of rows currently being skipped because their last run failed to write a hemnet_listingv2 row.

## What this means for downstream plans

- **Plan 09-04 (Cutover + runbook):** the runbook should include "if a manual cron run shows `status='running'` for an unusual time, suspect SIGHUP" as a diagnostic step. Also: always run manual crons in tmux/nohup, never naked in a web console.
- **Mon 2026-05-25 03:00 UTC cron (next scheduled Hemnet match cohort firing):** runs against the freshly delta-filtered scope. Expected wall-clock 40-50 min (mirrors the W20 recovery rerun). If it lands closer to 70 min, the per-Oxylabs-call overhead is closer to the worst case and Phase 10's worker-pool refactor becomes more interesting.
- **The delta filter is now load-bearing.** If the cron_job_log table is ever wiped (rare but possible in dev/staging restore scenarios), the delta filter falls back to `MAX('2000-01-01'::timestamptz)` and effectively becomes a no-op (full scope). Acceptable degradation.
