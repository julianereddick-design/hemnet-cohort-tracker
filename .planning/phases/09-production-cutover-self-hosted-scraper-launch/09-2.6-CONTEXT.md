<domain>
## Plan Boundary

**Make Hemnet match cohort (`hemnet-targeted-match.js`) routinely finish in 60–90 min so it always completes before Cohort create's Mon 06:00 UTC checkpoint — AND retroactively rebuild W20 cohort_pairs to ~1,500 (vs the 441 partial cohort built tonight).**

This plan addresses the failure mode that surfaced on Plan 09-2.5's first wet cron grid (Sun 2026-05-17 → Mon 2026-05-18):

**The failure.** Plan 09-2.5's narrowed-search rewrite shipped and deployed successfully. Sun 22:00 UTC Booli fetch cohort completed (1,689 new + 595 updated W20 rows). Mon 03:00 UTC Hemnet match cohort started at 03:00:02 UTC and was still running 5+ hours later when Cohort create fired at 06:00 UTC. At 06:00 UTC, only ~200 of 5,563 W20 Booli rows had been processed, so W20 cohort_pairs ended up at 441 vs the ~1,500 typical for the week.

**Two compounding causes:**

1. **Per-row latency × conc 2.** `hemnet-targeted-match.js:680` uses `Promise.all([worker(), worker()])` — only 2 concurrent workers. Every search + detail fetch goes through Oxylabs Advanced (~5s/call) because the droplet IP is 100% blocked from Hemnet direct path (last night's run logged `oxylabsFallbackRate: 1`). Empirical throughput on the live run: **2.2 rows/min**. Theoretical max at conc 2 + Oxylabs is ~6 rows/min, so observation is roughly aligned with the Oxylabs latency floor.

2. **Over-broad SELECT scope.** `hemnet-targeted-match.js:609-620`'s WHERE clause is `is_active = true AND is_pre_market = false AND listed BETWEEN $1 AND $2 AND county = ANY(BOOLI_COUNTIES)` — every active W20-listed Booli row, no delta filter. Tonight's scope was 5,563 rows, ~3,100 of which already had valid `hemnet_listingv2` matches from prior weekly cycles. The script re-matches them every week, wasted work. Already flagged as STATE.md carry-forward `09-2.5 #11`.

**The architectural shift this plan locks:** raise concurrency to 8 (mirroring what Plan 09-02 D-15 successfully did for Booli view data / Job D), AND restrict the SELECT to "rows that genuinely need a match attempt this week." Together these compress the cron run from 30+ hours to ~50 min. Closes 09-2.5 #11.

Out of scope: investigating WHY the droplet IP is 100% blocked from Hemnet/Booli direct paths (operator decision 2026-05-18: accept Oxylabs as steady-state, don't spike IP rotation / UA changes / VPN). Restructuring the cron grid to every-2-days cadence (that's Plan 09-03's territory, untouched by this plan). Adjusting `cohort-create.js` matching SQL (separate concern; outside this plan). Hardening Job A (Hemnet view data) discovery-metadata overwrite with COALESCE-preserve (filed as STATE.md `09-2.5 #9` for Phase 10).

<decisions>
## Implementation Decisions

### Throughput (D-32 .. D-34)

- **D-32:** Bump `hemnet-targeted-match.js` worker pool from concurrency 2 to concurrency 8. Mirror the exact pattern Plan 09-02 D-15 established for `booli-targeted-refresh.js:323`:
  ```js
  await Promise.all(Array.from({ length: 8 }, () => worker()));
  ```
  Replaces `Promise.all([worker(), worker()])` at `hemnet-targeted-match.js:680`. No other code changes required — the worker function, jitter, search cache, in-flight dedupe, and per-row try/catch all already exist and scale to higher concurrency.
  - **Rationale:** Job D's identical conc-2→8 bump in Plan 09-02 was validated empirically; same Oxylabs Advanced credentials, same retry logic. Expected speedup: ~4× (6 rows/min → ~24 rows/min on the search-only path; ~12 → ~48 on a typical mixed match/miss path).

- **D-33:** Add a delta filter to the SELECT so re-runs and first-runs of an unchanged week skip already-matched rows. Filter on `booli_listing.crawled` against the timestamp of the last successful Hemnet match cohort run:
  ```sql
  AND (
    bl.crawled > (
      SELECT COALESCE(MAX(started_at), '2000-01-01'::timestamptz)
        FROM cron_job_log
       WHERE script_name = 'hemnet-targeted-match'
         AND status IN ('success', 'warning')
    )
    OR NOT EXISTS (
      SELECT 1 FROM hemnet_listingv2 h
       WHERE h.is_active = true
         AND h.postcode = bl.postcode
         AND LOWER(TRIM(h.street_address)) = LOWER(TRIM(bl.title))
         AND h.listed BETWEEN bl.listed - INTERVAL '7 days'
                          AND bl.listed + INTERVAL '7 days'
    )
  )
  ```
  Logic: process a row IF it was touched in `booli_listing` since the last Hemnet match cohort run (new data to match) OR if no `hemnet_listingv2` row currently satisfies cohort-create's matching predicate (we still don't have a match). Skip everything else.
  - **Rationale:** Tonight's bucketing showed 2,385 of 5,563 W20 rows were touched since Sun 22:00 — those need re-matching. The other ~3,100 are stable; if they're already matched in `hemnet_listingv2`, no need to re-fetch from Hemnet.
  - **Trade-off considered & rejected:** Filtering against `cohort_pairs` directly (the obvious "skip already-paired" filter) doesn't work because `cohort_pairs` is built BY `cohort-create.js` AFTER Hemnet match cohort. On the first run of a week, cohort_pairs is empty for that cohort_id → no rows skipped → no speedup. The `crawled` + `hemnet_listingv2 NOT EXISTS` shape works on first AND repeat runs.
  - **Edge case:** the EXISTS subquery cost. The `hemnet_listingv2` JOIN on `LOWER(TRIM(...))` is the same predicate `cohort-create.js:116-127` uses; if there's a functional index it's fast. If not, ~5,563 EXISTS evaluations × ~100k hemnet_listingv2 rows could be slow on the SELECT itself. Validate with EXPLAIN before merging; if slow, fall back to the simpler `bl.crawled > <last-run>` filter alone (loses some re-match-where-needed coverage but still cuts scope by ~57%).

- **D-34:** Add `JOB_BUDGET_MS = 120 * 60 * 1000` wall-clock budget + `validate()` branch (currently absent — the script runs forever if pathologically slow). Mirror Job D's pattern at `booli-targeted-refresh.js:40, 289-294, 361-362`:
  - Top of each worker iteration: check elapsed time; if > budget, set `summary.budgetExceeded = true`, log warning, drain queue, break.
  - `validate(summary)`: if `budgetExceeded === true`, return `job budget exceeded (${JOB_BUDGET_MS}ms); ${summary.matchedFromSearch} matches before drain`.
  - **Sizing rationale:** Expected runtime post-D-32/D-33 is ~50 min on a typical week (2,400 rows / 48 rows/min). 120-min budget gives ~70 min margin. Cron fires Mon 03:00 UTC → completes by Mon 05:00 UTC → 60 min margin before Cohort create's 06:00 UTC. Plenty of headroom.

### Operating assumptions (D-35)

- **D-35:** **Accept 100% Oxylabs as the steady-state for Hemnet match cohort.** Don't spike droplet IP rotation, User-Agent rotation, or VPN paths. The Oxylabs Advanced cost (~$0.005/call × ~3,000 calls/week from this script = $15/week) is acceptable. Per-call latency budget is the constraint that informs D-32 (conc 8) and D-34 (120 min budget).
  - **Operator decision (2026-05-18):** explicit pushback against scoping an IP-rescue investigation into this plan. Aligns with the operator preference for tactical fixes over hopeful infrastructure spikes.

### W20 recovery (D-36 .. D-37)

- **D-36:** **W20 recovery is Task 5 of this plan**, not a floating operator task afterward. Treats the recovery as a first-class deliverable with its own acceptance criteria + paper trail.

- **D-37:** **Run W20 recovery from the droplet, not from local.** Despite local IP not being blocked from Hemnet/Booli direct, recovery sequence runs from droplet because:
  - Laptop IP changes every ~3 hours killed Friday's enrichment and tonight's Skåne/Uppsala catch-up runs mid-execution (had to re-whitelist via doctl and observe partial completion)
  - Droplet IP is stable across the 50-min recovery window
  - Post-D-32 + D-33, droplet at conc 8 + Oxylabs should hit the same throughput we'd get from local at conc 8 — the new code is the speedup, not the environment.
