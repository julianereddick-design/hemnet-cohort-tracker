<domain>
## Plan Boundary

**Take over Booli-side enrichment fields that the Django scraper was supplying (until its 2026-05-15 decommission), AND rewrite Job B's Hemnet seeding to use those new fields as targeted-search filters — closing the 26.7% aggregate cohort match rate gap before Monday's cohort-create run.**

Two intertwined problems surfaced during the Plan 09-02 execution session that this plan addresses as a single piece of work:

**Problem 1 — Django scraper decommissioning.** The cohort tracker's `booli_listing` table has columns `price`, `agent_id`, `removed`, `currency` that our self-hosted Job C (`booli-targeted-discovery.js`) and Job D (`booli-targeted-refresh.js`) never populate. Historical data shows these columns were ~75-95% populated by a daily ~07:00 UTC writer that is not in this repo — confirmed by `booli-targeted-discovery.js:21` header comment "safe against parallel Django writes." On 2026-05-15, that writer abruptly dropped from ~75% population to 1/610 — the Django scraper is in mid-decommission. Phase 9 ROADMAP says "both external scrapers already off" as the end-state; today's drop is that promise being kept. Without us capturing these fields, Metabase reports keyed on them silently degrade and no Booli-side discriminator data exists for the Hemnet matching strategy below.

**Problem 2 — Hemnet matching is much worse than thought.** Cohort-create's matching SQL (`cohort-create.js:116-127`) uses exact `postcode = $1 AND LOWER(TRIM(street_address)) = LOWER(TRIM($2)) AND listed BETWEEN $3 - 7d AND $3 + 7d`. Project memory + ROADMAP carry-forward refers to a 42.4% Hemnet match rate (VERF-05 snapshot). Live query (2026-05-15) over the last 8 cohort weeks shows **26.7% aggregate match rate** (5484 matched / 15036 unmatched) — substantially worse. Operator hypothesis (confirmed in session): **Job B is the dominant bottleneck — it doesn't FIND the right Hemnet listings.** Today Job B paginates 15 pages of Hemnet's location-based search per Booli municipality (~750 cards per muni walk) and filters in-app by `streetAddress + ±7 days`. Listings outside that 15-page window, or whose street string differs slightly from Booli's, are silently dropped. Even if we fixed cohort-create's matching SQL, the Hemnet row often isn't in the DB to match against.

**The architectural shift this plan locks:** Job B switches from "paginate everything in a muni, post-filter by address" to "query Hemnet's search directly with Booli's `price + rooms + object_type` as URL filters, narrowing to ~3-10 candidates per Booli row." That's only possible if Job B can READ Booli's `price`, `rooms`, `object_type` — which is why field capture is a prerequisite, not a separate concern. The plan ships them as one cohesive change.

Out of scope: changing cohort-create's matching SQL (the matched-pair correctness audit will say whether this is needed), `cohort_unmatched` bucket-level diagnostic (DB queries time out — NOT EXISTS scans against 100k+ hemnet_listingv2 rows without indexed `LOWER(TRIM(...))`), backfill of price/rooms onto historical rows (Job D's COALESCE-preserve UPDATE will organically backfill over 1-2 cron cycles), Booli detail-page schema changes for fields we don't need now (coordinates, price history, image arrays).

</domain>

<decisions>
## Implementation Decisions

### Booli field capture (D-20 .. D-24)

- **D-20:** Capture **five fields** from Booli detail pages going forward (Job C insert + Job D refresh):
  - `price` (int SEK) — Hemnet match filter primary discriminator + Metabase reports
  - `rooms` (numeric, nullable) — Hemnet match filter secondary discriminator (Booli sometimes returns 2.5)
  - `living_area` (numeric m², nullable) — match confirmation field + Metabase reports
  - `object_type` (varchar, Swedish string e.g. `Lägenhet`/`Villa`) — Hemnet match filter via mapping table
  - `agent_id` (bigint) — Metabase broker reports continuity
  - **Rationale:** Minimum set that (a) enables Hemnet targeted search, (b) preserves Metabase report continuity post-Django, (c) avoids schema bloat. Coordinates, price history, image URLs all deferred.
  - **Skipped:** `images` (count-only, meaningless without URLs), `removed` (redundant with `is_active=false` flag from Job D).

- **D-21:** Apollo state key paths (verified 2026-05-15 via live detail-page spike against 3 random active URLs):
  | Field | Apollo path | Coercion |
  |---|---|---|
  | price | `Listing.listPrice.raw` | int or null |
  | rooms | `Listing.rooms.raw` | numeric or null (allows 2.5) |
  | living_area | `Listing.livingArea.raw` | numeric or null (m²) |
  | object_type | `Listing.objectType` | string or null |
  | agent_id | `apolloState[Listing.source.__ref].id` parseInt | bigint or null |
  - **Implementation:** `lib/booli-fetch.js:69-150` `parseBooliListing` extended; smoke fixture updated. Every field coerces to null on path miss — no crash on malformed Apollo state.

- **D-22:** **`agent_id` SEMANTIC DIVERGENCE — flagged for Metabase consumers.**
  - Booli's `Listing.source.__ref` resolves to `Source:N` where N is the **broker chain ID** (e.g. `Source:64` = Länsförsäkringar Fastighetsförmedling, a national chain).
  - Django's old `agent_id` values are 4-9 digits and don't match Source.id for the same booli_id. Django was capturing from a different source/endpoint (possibly individual agent ID, not chain).
  - **Going-forward semantics:** our `agent_id` column will hold Source.id (broker chain). Some are 1-4 digit (chain IDs), some are 9-digit (possibly individual agent IDs that Booli's Source object exposes for chain-less listings).
  - **Action item for operator:** Metabase reports/dashboards keyed on Django agent_id values (counts, joins, filters) may need rebuilding against chain-level granularity instead of agent-level.

- **D-23:** **Schema migration** — add 3 new columns to `booli_listing` (run live 2026-05-15 via `migrate-booli-listing-add-fields.js`):
  ```
  rooms        NUMERIC(3,1) NULL
  object_type  VARCHAR(40)  NULL
  living_area  NUMERIC(6,1) NULL
  ```
  `price` (integer) and `agent_id` (bigint) columns pre-existed. All NULL-able; no backfill. Idempotent (`ADD COLUMN IF NOT EXISTS`).

- **D-24:** **COALESCE-preserve write semantics** in both Job C INSERT/ON CONFLICT and Job D UPDATE: every new field is written as `COALESCE(EXCLUDED.x, booli_listing.x)`. Reason: if Booli's Apollo state is briefly malformed on a refresh, we don't silently clobber a previously-good value with null. Tradeoff: stale values persist until next clean fetch. Acceptable; the alternative (let null wins) was the failure mode that bit Plan 09-01.

### Hemnet item_types[] vocabulary + mapping (D-25)

- **D-25:** **Booli → Hemnet object_type mapping** (verified 2026-05-15 via live probe of `hemnet.se/bostader?item_types[]=<token>` GraphQL search args):
  | Booli (8 observed types in 140-card cross-county sample) | Hemnet URL token | Hemnet internal group |
  |---|---|---|
  | `Lägenhet` (53%) | `bostadsratt` | `APARTMENTS` |
  | `Villa` (34%) | `villa` | `HOUSES` |
  | `Radhus` (5%) | `radhus` | `ROW_HOUSES` |
  | `Fritidshus` (4%) | `fritidshus` | `VACATION_HOMES` |
  | `Kedjehus` (2%) | `villa` | `HOUSES` (Hemnet bundles) |
  | `Parhus` (would-map) | `villa` | `HOUSES` (Hemnet bundles) |
  | `Tomt/Mark` (1%) | `tomt` | `PLOTS` |
  | `Gård` (1%) | `gard` | `HOMESTEADS` |
  - **All 140 sampled Booli `objectType` values have a defined mapping target** — no orphan types in the dominant 4-county Booli FS inventory.
  - **Hemnet IGNORED tokens** (verified): `kedjehus`, `parhus`, `lagenhet`, `andelsbostad`, `agarlagenhet`, `tomt-mark` (my initial guess; correct is `tomt`), `gard-skog` (my initial guess; correct is `gard`). The IGNORED tokens cause Hemnet to silently disable the filter, returning all listings. Our mapping deliberately avoids them.
  - **Implementation:** `lib/booli-to-hemnet-mapping.js` (NEW). Exports `BOOLI_TO_HEMNET_ITEM_TYPE` frozen object + `booliObjectTypeToHemnet(s)` helper that returns null on unknown. Smoke: 14 pass.

### Job B targeted-search rewrite (D-26 .. D-29) — PENDING IMPLEMENTATION

- **D-26:** **Job B's seeding query switches from location-paginated to Booli-discriminator-filtered.** Current Job B at `hemnet-targeted-match.js:160` walks `?areaIds={muni}&page=N` for up to MAX_PAGES=15. New Job B walks Hemnet's `/bostader` filtered by Booli's known data:
  ```
  https://www.hemnet.se/bostader
    ?location_ids[]={hemnet_muni_id}
    &price_min={booli.price * 0.95}
    &price_max={booli.price * 1.05}
    &rooms_min={booli.rooms}
    &rooms_max={booli.rooms}
    &item_types[]={booliObjectTypeToHemnet(booli.object_type)}
  ```
  Expected candidate set per Booli row: **3-10 cards** vs. today's ~750. Job B then fetches details for that narrow set and runs the existing `cardMatches` predicate (street + ±7 days) as confirmation. Same INSERT/UPDATE contract into `hemnet_listingv2`.
  - **Rationale:** The narrowing math: 750 cards × ~50% with broadly-similar room counts × ~30% in similar price band = ~110 candidates broadly. Tightening rooms to exact match cuts that ~5x. Tightening price to ±5% cuts ~3x. Adding item_types[] filter cuts another ~2x. Final ~3-10 — small enough to fetch detail-page for ALL candidates and run secondary confirmation.
  - **Trade-off vs. status quo:** Today Job B has location-search budget cost ~15 pages × N munis ≈ 60 search calls. New Job B has ~1 search call per Booli row but ~5-10 detail-fetch calls per Booli row (vs. today's ~1-2 successful matches). Total Oxylabs cost ≈ similar; coverage rate ≈ much higher.

- **D-27:** **Price tolerance band: ±5%.** Booli's `listPrice.raw` and Hemnet's published asking price should agree exactly for the same listing, BUT one side may have observed a price reduction the other hasn't yet. ±5% gives slack for the lag without admitting too many wrong-listing candidates. Revisit if false-positive rate is high.
  - **Open question for impl review:** does ±5% need rounding to nearest 50k for Hemnet's URL semantics? Hemnet's price filter snaps to round numbers in its UI. Test on first wet-run.

- **D-28:** **Rooms tolerance: exact (min=max).** Setting `rooms_min=rooms_max=N` keeps Hemnet returning only exactly-N-room listings. For Booli's decimal rooms (e.g. 2.5), set Hemnet to integer floor or round-to-nearest — TBD on impl. Most listings are integer.

- **D-29:** **Fallback when filter values missing.** When Booli row has `price=null` or `rooms=null` or `object_type=null` (e.g. just inserted by Job C with bad Apollo state, or pre-D-23 legacy rows), Job B drops the missing filter and broadens the search. Order of fallback: drop item_types[] first (smallest narrowing benefit), then price, then rooms. Never drop location_ids — that's load-bearing.
