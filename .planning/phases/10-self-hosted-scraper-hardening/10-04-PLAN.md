## 10-04 (a) — `export-views-wide.js` gap-aware delta (the only functional item)

**Problem:** `export-views-wide.js` builds three CSVs (cumulative / incremental / rolling-7d) + an aggregate. The **1-day incremental** section assumes consecutive tracked dates are exactly 1 day apart — `export-views-wide.js:159-160`:
```js
const gap = Math.round((currDate - prevDate) / DAY_MS);
if (gap !== 1) continue;
```
Under the post-cutover every-2-days cohort-track cadence, consecutive snapshots are ~2 days apart, so `gap !== 1` is true almost always and the 1-day incremental columns come out blank for W21+. Confirmed in the 2026-05-26 W20 export: the aggregate 1d-section had values only for 2026-05-21 and was blank for 2026-05-23 / 2026-05-25.

**Note** the same file already handles Booli's every-other-day cadence correctly in the 7-day rolling section and in a 2-day Booli delta (`export-views-wide.js:171-182`) — so the rolling-7d output is fine. Only the 1-day incremental file is stale.

**Fix options:**
- Replace the `gap === 1` assertion with a gap-aware daily rate: `(curr - prev) / gap` so a 2-day gap yields a per-day average. Mirrors the existing Booli 2-day-average logic already in the file.
- OR drop the 1-day incremental file entirely and rely on the 7-day rolling file (which is what the hb-ratio xlsx + charts already use as the canonical metric).

**Recommend** the gap-aware approach — keeps the file useful, ~20 lines around `export-views-wide.js:149-188`. Also check `export-hb-ratio-xlsx.js` — it uses a fixed 2-day lookback (`SEC4` incremental section + the chart data at lines 488-514) which is intentional for Booli and unaffected, but confirm it doesn't have the same 1-day assumption anywhere.

**Verification:** `node export-views-wide.js --cohort 2026-W21 --include-latest` and confirm the incremental CSV now has populated 1-day columns. Compare totals against the cumulative CSV by hand for one pair.

## 10-04 (d) — Final-JSON `status` vs cron-wrapper status mismatch (09-01 #3)

**Problem:** several scrape scripts emit a `log('INFO', 'Final: ' + JSON.stringify(summary))` line where `summary` may carry a `status` field that says `success`, while cron-wrapper independently writes `status='warning'` (or `failure`) to the `cron_job_log` row based on `validate()`. The two disagree, which is confusing when grepping `/var/log/hemnet/*.log` against the DB.

**Fix:** either drop the `status` key from the in-job `summary` object (let cron_job_log.status be the single source of truth), or have the job read back the resolved status. Dropping is simpler. Find the offending `Final:` emissions (grep `Final:` across the scrape scripts) and remove the redundant status field. ~5 lines per script, cosmetic.

**Verification:** grep `Final:` log lines after a run; confirm no `status` key, or that it matches the DB row.
