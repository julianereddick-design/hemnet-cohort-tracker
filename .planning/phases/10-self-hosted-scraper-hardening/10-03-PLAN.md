# Plan 10-03: cohort-track null-Booli warning retarget

## Context (read this cold — it has everything a fresh session needs)

`cohort-track.js` runs every 2 days (Tue/Thu/Sat/Mon-ish, 22:00 UTC) and snapshots each active cohort pair's Hemnet + Booli view counts into `cohort_daily_views`. At the end of each run its `validate()` callback emits a Slack `warning` if **any** cohort has more than 50% of its pairs missing fresh Booli (or Hemnet) view data.

**The exact code** — `cohort-track.js:272-288`:
```js
validate: (summary) => {
  if (summary.totalTracked === 0 && summary.cohortsTracked > 0) {
    return `0 pairs tracked across ${summary.cohortsTracked} active cohort(s) — expected hundreds`;
  }
  const warnings = [];
  if (summary.perCohortNull) {
    for (const c of summary.perCohortNull) {
      const bPct = Math.round((c.nullBooli / c.tracked) * 100);
      const hPct = Math.round((c.nullHemnet / c.tracked) * 100);
      if (bPct > 50) warnings.push(`${c.cohortId}: ${bPct}% null Booli (${c.nullBooli}/${c.tracked})`);
      if (hPct > 50) warnings.push(`${c.cohortId}: ${hPct}% null Hemnet (${c.nullHemnet}/${c.tracked})`);
    }
  }
  if (warnings.length > 0) return warnings.join('; ');
  return null;
},
```

**The problem (proven during the Phase 9 green-week observation):** the >50% threshold fires on EVERY run because old cohorts naturally accumulate null Booli as their listings sell or withdraw from Booli's active feed. The every-2-days Booli view-data refresh only touches listings still active on Booli — once a property goes off-feed, that pair is permanently null going forward.

The green-week data showed this is structural, not a recovery problem (the original 09-03 #4 hypothesis that it would "self-clear" was wrong):

| cohort-track fire | W14 null-Booli | W15 | W16 | W20 | W21 (newest) |
| ----------------- | -------------- | --- | --- | --- | ------------ |
| 2026-05-21 22:00  | 59%            | 54% | 49% | 4.8% | —           |
| 2026-05-23 22:00  | 60%            | 56% | 51% | 8.5% | —           |
| 2026-05-25 22:00  | 61%            | 56% | 51% | 10.5% | 1.0%        |

Old cohorts (W14-16) only climb. The newest cohort (W21) is healthy at 1%. So the warning is permanent Slack noise for any cohort more than ~3-4 weeks old, which drowns out the signal it was meant to provide (detecting a genuine Booli-refresh outage on the cohorts we still actively care about).

## Decision needed (ask the operator FIRST — do not pick unilaterally)

Three retargeting strategies. Each defines "interesting" differently:

- **(a) Age-bounded threshold.** Only apply the >50% check to cohorts ≤ N weeks old (e.g. N=2 or 3). Rationale: a high null rate on a *fresh* cohort means the Booli refresh genuinely broke this week; on an old cohort it's just natural decay. Simple, preserves the original alerting intent for the cohorts that matter. **Recommended starting point** — pair with whatever N matches how long Booli keeps a typical listing on its active feed.
- **(b) Week-over-week delta threshold.** Warn only if a cohort's null pct jumped by more than X percentage points since the previous run. Catches genuine breakage on any cohort, ignores gradual decay. More robust but needs to persist/recompute the prior run's per-cohort null pct (read from the previous cron_job_log row's result_summary, or recompute). More code.
- **(c) Demote to reporting field.** Drop the warning entirely; keep per-cohort null pct in `result_summary` (it's already there as `perCohortNull`) for dashboards/manual inspection. Simplest. Rely on the per-scrape-job Oxylabs/error checks (Booli view data's own `validate()`) to catch a refresh outage instead. Loses the cohort-level cross-check.

**Open question to resolve the decision:** does the operator want an *active alert* when the current week's Booli refresh breaks, or is the per-job (Booli view data) alerting enough? If the per-job alerting is trusted, (c) is cleanest. If a cohort-level cross-check is wanted, (a) with N≈2-3 weeks is the move.

## Implementation sketch (once the option is chosen)

All three are edits to the single `validate()` block at `cohort-track.js:272-288`. No schema change, no migration.

- **(a)** needs a way to compute cohort age. `summary.perCohortNull[].cohortId` is an ISO week label like `2026-W21`. Compare against the current ISO week (the newest cohort_id, or derive from `new Date()`). Add `if (weeksOld > N) continue;` before the threshold check. Watch the ISO-week-arithmetic edge cases (year boundaries) — reuse the `getCohortWeek`-style day-of-year math already in `cohort-create.js:17-44` / `booli-targeted-discovery.js:169-196` rather than rolling new week math.
- **(b)** read the prior run's `result_summary.perCohortNull` from `cron_job_log` (most recent `cohort-track` row before this one) and diff. Needs a DB read inside `validate()` or passed through `main()`'s summary.
- **(c)** delete the `bPct > 50` / `hPct > 50` push lines; keep building `perCohortNull` in the summary so it's still recorded.
