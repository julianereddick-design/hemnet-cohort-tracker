<objective>
Build `scripts/sold-match-run.js` ‚Äî the one config-driven, manually-runnable end-to-end runner that replaces the throwaway `scripts/spike-hemnet-match.js`. Per configured segment (from `config/sold-segments.json`) and a rolling sold-date window (default ~30-day monthly window ending at the 90-day `READ_TIME_EXCLUDE_DAYS` boundary), it: seeds Booli via `fetchBooliSoldPage` + `upsertBooliSold`, searches Hemnet `/salda` per record via `searchSoldPaged`, adjudicates each non-deed-transfer record (apartments fee-exact, villas address-key), and persists a verdict via `persistVerdictForRecord` ‚Äî under the Phase-16 DB-atomic spend ceiling with a bounded ~6-worker pool. It prints a per-segment run summary and writes NO report file.

This is ORCHESTRATION WIRING. Every algorithm already exists; the runner assembles them. Mirror the spike's `matchOne` / worker-pool shape exactly, with three deliberate divergences:
1. **DB persist** replaces JSONL writes (`upsertBooliSold` / `upsertHemnetSold` / `persistVerdictForRecord`).
2. **Drop the recall pass** (D-03) ‚Äî non-matched records emit `verdict = booli_only` with no second loose search.
3. **Inline apartment fee fetch** via the exported `fetchBooliDetail` (D-06 fix) ‚Äî seed-time `rent` is null for the monthly window.

Purpose: Closes MATCH-01, MATCH-03, MATCH-04, CONFIG-02 ‚Äî the runnable pipeline that turns the validated spike logic into a DB-backed tool.
Output: `scripts/sold-match-run.js` (new), with an offline `--smoke` block.
